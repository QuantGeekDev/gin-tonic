New OpenClaw Architecture Kata

A Monorepo-First Guide for Shared CLI + Web Agent Runtime

Why This Version Exists

Your project already has two channels:

  • CLI operator console (apps/cli)
  • Web dashboard + API (apps/web)

The first kata version assumed a single CLI app first, then gateway later.
This revised kata matches your current reality: multi-channel from the start, with one shared runtime so behavior does not drift.

Core rule for this kata:

Channels are adapters. Runtime is shared. Session state is server-owned.

Kata Structure (Revised)

Build runAgentTurn once in packages/agent-core.
Both CLI and web use it.

Deliverables:

  • packages/agent-core/src/agent/loop.ts
  • packages/agent-core/src/types.ts
  • packages/agent-core/src/config/agent.ts
  • tool execution callback contract

Acceptance:

  • Same prompt + same tool input => same assistant/tool behavior in CLI and web.

Part 2: JSONL Sessions + Shared Message Handler

This is the most important architecture shift in the kata.

Goal: move from “UI owns conversation state” to “shared runtime owns conversation state”.

Why JSONL, in plain terms

Use one file per session.
Each line is one JSON object (one message).

Benefits:

  • append-only writes are simple and crash-tolerant
  • easy to inspect with terminal tools
  • no DB setup needed while learning architecture

Mental model:

  • session file = transcript log
  • each new message = append one line
  • compaction (later) = rewrite file deliberately

JSONL rules to follow

  1. One complete JSON object per line.
  2. Always end appended lines with \n.
  3. Ignore empty lines when reading.
  4. If a line fails JSON parse, skip it and continue (graceful recovery).
  5. Never partially rewrite the file during normal turns.

Part 2.1: Define session locations and ownership

Create these shared runtime files:

  • packages/agent-core/src/sessions/store.ts
  • packages/agent-core/src/sessions/keys.ts
  • packages/agent-core/src/gateway/handle-message.ts

Decide where files live on disk (example):

  • .data/sessions/<sanitized-session-key>.jsonl

Do this now:

  • choose one base sessions directory
  • keep it outside app-specific folders so CLI/web both hit the same store

Checkpoint:

  • both channels can resolve the same absolute sessions directory.

Part 2.2: Build session key builder first

In sessions/keys.ts, define one function that always builds keys the same way.

Inputs:

  • agentId
  • scope (channel-peer, peer, global)
  • channelId
  • peerId

Output format (stable string):

  • agent:<agentId>:scope:<scope>:peer:<peerId>:channel:<channelId>

Also add sanitization for filesystem safety:

  • replace /, :, \\, whitespace with _

Checkpoint:

  • same inputs always produce same key
  • “dangerous” characters are neutralized

Part 2.3: Implement read path (load) in SessionStore

Start with read before write.

Behavior:

  • if file does not exist, return empty message list
  • read line-by-line
  • trim line
  • skip empty lines
  • try parse JSON
  • validate shape is a Message
  • skip malformed lines, continue

Why this order:

  • you can safely introspect existing data before mutating anything

Checkpoint:

  • loading an empty/missing session returns []
  • loading mixed good/bad lines returns only good messages

Part 2.4: Implement append path (append) in SessionStore

Then add append-only writes.

Behavior:

  • ensure parent directory exists
  • serialize one message as JSON
  • append exactly one line plus newline

Important:

  • do not read-modify-write for normal turns
  • append is the default write mode

Checkpoint:

  • appending twice creates two lines
  • file stays valid JSONL

Part 2.5: Add full rewrite path (save) for future compaction

You need a controlled rewrite function, but only for explicit operations (like compaction).

Core rule for this step: write to the persistent canonical session file path directly. Do not use temporary files for this kata step.

Behavior:

  • overwrite the persistent session file with provided messages
  • write one JSON object per line (JSONL)
  • keep deterministic ordering from the input array
  • no temp file workflow; no temp-to-rename flow in this step

Checkpoint:

  • save produces deterministic line order
  • load(save(messages)) round-trips correctly
  • the canonical session file path is the only file touched

Part 2.6: Create shared handleMessage orchestrator

Core problem:

Today, channel adapters can drift into owning conversation state rules. That creates inconsistent behavior between CLI and web, and makes bugs hard to reason about.

Approach:

Create one shared orchestrator in packages/agent-core/src/gateway/handle-message.ts. Every channel calls it. This centralizes routing, session state, model execution, and persistence in one place.

Didactic step-by-step pipeline:

  1. Normalize inputs: resolve effective routing info (agentId, scope, channelId, peerId). Explain in code comments that this guarantees one canonical routing context per request.
  2. Build session identity: construct a deterministic session key from normalized routing info. This is how all channels map to the same conversation.
  3. Load state from storage: read prior transcript messages from SessionStore using that key.
  4. Apply current user event: append the incoming user message in memory to form the turn input.
  5. Execute domain logic: call runAgentTurn with the assembled message list.
  6. Persist resulting state: write new transcript state through SessionStore (append for normal turns, save for explicit full-rewrite operations).
  7. Return contract output: respond with assistant text plus session metadata needed by adapters.

Channel adapters call this; they should not own transcript logic.

Adapters:

  • CLI: apps/cli/src/ui/app.tsx (or a thin adapter module)
  • Web: apps/web/app/api/agent/route.ts

Validation:

  • Unit test each stage boundary: routing normalization, session-key determinism, store interaction, and response mapping.
  • Integration test one end-to-end turn through handleMessage with mocked runAgentTurn.
  • Cross-channel test: same logical input via CLI and web produces equivalent persisted transcript behavior.

Why this is enterprise-grade:

  • single orchestration path reduces channel drift and operational surprises
  • deterministic session identity improves auditability and incident debugging
  • clear boundaries (adapter vs core) support safer scaling and future channels
  • testable stage contracts enable reliable refactors under change

Checkpoint:

  • same input through CLI or web hits same shared pipeline.

Part 2.7: Change web contract to server-owned sessions

Current stepping stone may still send messages from browser. For Part 2 completion, stop that.

Preferred request shape:

  • peerId
  • text
  • optional agentId
  • optional scope

Server does:

  • load prior messages from session store
  • run turn
  • persist

Checkpoint:

  • browser refresh does not lose session continuity
  • web and CLI can continue one shared conversation (same peerId + scope)

Suggested test sequence (small and safe)

  1. Unit test buildSessionKey deterministic + sanitization behavior.
  2. Unit test SessionStore.load with malformed lines.
  3. Unit test SessionStore.append line count and newline correctness.
  4. Unit test SessionStore.save round-trip with load.
  5. Integration test handleMessage with mocked runAgentTurn.
  6. Cross-channel test: CLI turn then web turn on same session key.

Acceptance for Part 2:

  • CLI says: “My name is Alex.”
  • Web (same peerId scope) asks: “What is my name?”
  • Assistant answers: “Alex.”
  • Session file shows append-only JSONL transcript.

Migration Plan from Current Codebase

  1. Keep loop/types/defaults in packages/agent-core (already done).
  2. Add sessions/* and gateway/handle-message.ts in packages/agent-core.
  3. Move session ownership from app state to shared handler.
  4. Change web request contract to { peerId, text, agentId, scope } and stop sending full messages.
  5. Keep CLI and web as presentation/adapters only.

Definition of Done

You are done when:

  1. No app imports runtime files from the other app.
  2. Shared runtime has tests for loop, config, sessions, and handler.
  3. CLI and web can continue one conversation using shared session keys.
  4. Behavior/tool availability is consistent across both channels.

Practical Notes

  • Keep all shared contracts in one package (types.ts, tool definitions, session schema).
  • Prefer append-only JSONL for sessions; use overwrite only for compaction.
  • Add one cross-channel integration test early; it will catch drift faster than unit tests.

This kata path keeps your OpenClaw learning goals while matching your real architecture today.
