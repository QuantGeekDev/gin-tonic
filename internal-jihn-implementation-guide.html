Jihn Internal Implementation Guide

Scope: architecture kata implementation (Parts 1-8), MCP integration, refactors, validation strategy, and operator verification playbook.

Internal Monorepo CLI + Web Agent Core Shared Runtime

1. Architecture Summary

Core rule: channels are adapters; runtime is shared.

  • packages/agent-core contains session, gateway, compaction, memory, prompt composition, policy, routing, MCP integration.
  • apps/cli and apps/web call shared runtime and should not import each other.
  • Session state is server/runtime-owned and persisted by session key.

2. Kata Parts Implemented

Part 1: Shared Agent Loop

What we implemented: shared runAgentTurn and shared model/config defaults in agent-core.

Rationale: deterministic behavior parity between CLI and web.

Verify:

npm run test --workspace=packages/agent-core
npm run test --workspace=apps/cli

Part 2: JSONL Sessions + Shared Message Handler (2.1-2.7)

What we implemented: session key builder, session store load/append/save, shared gateway handleMessage, web server-owned contract.

Rationale: no browser-owned canonical transcript; shared persistence path for both channels.

Important: persistent session files are used (no temp-file strategy for save path in kata flow).

Verify:

  1. Use same peerId + scope=peer in web and CLI; confirm continuity.
  2. Set scope=channel-peer; confirm web and CLI diverge (isolated histories).

Part 3: Composable Prompts

What we implemented: prompt assembly in agent-core from AGENTS.md, SOUL.md, TOOLS.md and optional per-agent dirs.

Rationale: prompt governance via files, not hardcoded strings.

Verify: edit workspace prompt files and run both channels; behavior changes in both.

Part 4: Tool Policy + Permission Controls

What we implemented: shared policy executor with allow/deny/review modes and hooks.

Rationale: single policy behavior across channels.

Verify: deny a tool via env and confirm both web and CLI block it consistently.

Part 5: Context Compaction

What we implemented: token-budget compaction in packages/agent-core/src/sessions/compactor.ts with deterministic outcomes.

Rationale: keep model context bounded and channel-independent.

Verify: use web "Simulate Compaction" and compare web vs CLI identicalness output.

Part 6: Long-Term Memory

What we implemented: file-backed memory store with save_memory and memory_search, shared between channels.

Rationale: durable cross-session knowledge.

Verify: save memory in CLI interaction, search from web memory panel (and vice versa).

Part 7: Gateway Hardening

What we implemented: per-session lock manager, optional idempotency keys, structured gateway logs/errors.

Rationale: safe concurrent behavior and predictable retries.

Verify:

  • Concurrent same-session requests serialize.
  • Same idempotency key + same payload returns cached result.
  • Same idempotency key + different payload returns conflict.

Part 8: Multi-Agent Routing

What we implemented: command routing in agent-core for /research ... and /agent:<name> ....

Rationale: separate per-agent session histories/config with shared infrastructure.

Verify: send /agent:ops hello and observe session key agent segment changes.

3. Session Scoping Model

ScopePurposeBehavior
channel-peer (default)Safe channel isolationCLI and web do not share transcript
peerCross-channel continuityCLI and web share transcript for same peer
globalShared bot stateBroad shared state, use carefully

Canonical format: agent:<agentId>:scope:<scope>:peer:<peerId>:channel:<channelId>

4. MCP Integration (Added on Top of Existing Tool Model)

Design: local tools remain first-class; MCP tools are overlaid via namespaced tool IDs.

  • MCP servers config parser: packages/agent-core/src/mcp/config.ts
  • MCP registry and execution bridge: packages/agent-core/src/mcp/registry.ts
  • Type contracts: packages/agent-core/src/mcp/types.ts
  • Exposed MCP tool naming: mcp__<serverId>__<toolName>

MCP Transport Notes

  • Uses official MCP TypeScript SDK @modelcontextprotocol/sdk with Streamable HTTP client transport.
  • Supports MCP session headers and reconnect/session semantics through SDK transport.
  • SSE transport is deprecated in SDK docs; Streamable HTTP is the preferred path.

MCP Environment Configuration

JIHN_MCP_SERVERS=[
  {"id":"docs","url":"https://mcp.example.com/mcp","headers":{"Authorization":"Bearer ..."}}
]
JIHN_MCP_CACHE_TTL_MS=30000

MCP Debug Surface

  • API endpoint: GET/POST /api/mcp
  • Dashboard panel: MCP server status, refresh, error details, exposed tool mapping.

5. Refactors Completed

  1. Type extraction to semantic modules (runtime, web API contracts, CLI UI types).
  2. Web route validation extracted to apps/web/app/api/agent/validation.ts.
  3. Dashboard behavior split into hooks:
    • use-agent-session.ts
    • use-memory-debug.ts
    • use-mcp-debug.ts
  4. Removed duplicate CLI-local custom tool implementations to prevent drift; shared runtime is canonical.

6. Test & Verification Playbook

Automated

npm run test --workspace=packages/agent-core
npm run test --workspace=apps/cli
npx tsc -p apps/web/tsconfig.json --noEmit

Manual - Core

  1. Run web and CLI.
  2. Send same request in both using same peer and scope=peer; verify continuity.
  3. Switch to channel-peer; verify isolation.
  4. Use /research and /agent:<name>; verify session routing changes.

Manual - Policy

  1. Set JIHN_TOOL_POLICY_MODE=deny and tool list.
  2. Trigger blocked tool in web and CLI.
  3. Verify structured blocked behavior in both.

Manual - Compaction

  1. Set context token budget env vars.
  2. Create long conversation.
  3. Use dashboard "Simulate Compaction"; verify web/CLI compare result.

Manual - Memory

  1. Save memory via one channel.
  2. Search memory via the other channel.
  3. Verify persisted cross-channel retrieval.

Manual - MCP

  1. Configure JIHN_MCP_SERVERS with at least one reachable MCP endpoint.
  2. Load dashboard; open MCP Debug panel.
  3. Verify server status and exposed tools list.
  4. Invoke an MCP tool through agent flow (tool call should appear in activity log).
  5. Use MCP refresh button and verify snapshot timestamp/tool list updates.

7. Rationale Summary (Enterprise Angle)

  • Single runtime: minimizes channel drift and lowers operational risk.
  • Server-owned sessions: preserves integrity and auditability of transcript state.
  • Structured gateway errors/logs: improves observability and retry safety.
  • Policy centralization: enforces consistent trust boundaries.
  • MCP overlay model: extends capability without rewriting local tool model.
  • Hook/module split: improves debuggability and maintainability.
